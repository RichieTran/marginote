1. **Global conventions:**
You are building a Chrome Extension called Marginote. These conventions apply to every prompt in this series:

Stack: Vanilla JS (ES modules), HTML, CSS. No frameworks, no build step, no TypeScript. All Chrome APIs via Manifest V3.
Code style: Async/await over promises. Small focused functions. JSDoc comments on exported functions. No external runtime dependencies.
Data model versioning: All data stored in chrome.storage.local includes a top-level schemaVersion integer (start at 1). The storage utility exposes a migrate() function that runs on load and handles missing fields on older records by filling them with defaults rather than failing. When later prompts add new fields, bump schemaVersion and extend migrate().
Error handling: User-facing errors (save failures, quota exceeded, etc.) surface as small non-blocking toasts in the popup/dashboard and console.warn in content scripts. Never swallow errors silently. Wrap every chrome.storage call in try/catch.
Cascade rules (important, apply consistently):

Deleting a project prompts the user to choose: delete all its annotations, or move them to the "General" project. Default selection is "move to General."
Deleting a subgroup never deletes its annotations; they become ungrouped within the same project.


Storage: Use chrome.storage.local. Request the unlimitedStorage permission in the manifest so users aren't capped at 10MB.
Testing checkpoint: After each prompt, manually verify the feature on at least three different site types: a static article (e.g., a Wikipedia page), a dynamic SPA (e.g., a GitHub file view), and a news site with heavy CSS (e.g., nytimes.com). Note any failures in a TESTING.md file.
Styling: All injected content-script CSS is scoped with a marginote- class prefix and uses all: initial on root containers to avoid host-page style bleed.

2. **Highlighting and saving annotations**
Building on the existing Marginote extension, implement the core highlighting system in the content script.
Selection tooltip. When a user selects non-empty text on any webpage, render a small floating tooltip anchored just above the selection's bounding rect with a Save button. The tooltip is injected into a dedicated <div id="marginote-root"> appended to document.body, and uses Shadow DOM to isolate its styles from the host page. The tooltip dismisses when: (a) the user clicks outside it, (b) the selection is cleared, or (c) Escape is pressed. Clicking Save saves the annotation via the storage utility and removes the tooltip.
Anchor strategy. When saving, capture:

selectedText — the selected string.
contextBefore / contextAfter — 30 chars before and after the selection in the container's textContent.
containerSelector — a selector for the nearest stable ancestor. Use a simple, pragmatic strategy: walk up from the selection's common ancestor and pick the first ancestor with an id (use #id); otherwise build a path of tagName:nth-of-type(n) up to body with a max depth of 6. Keep this in a single buildSelector(node) helper function. Do not pull in an external library.
textOffset — the character offset of the selection start within the container's textContent.

Re-highlighting on load. On page load (and on SPA route changes — use a MutationObserver on document.body with a debounce of 300ms to detect significant DOM changes), load all annotations for the current URL and re-apply highlights. Resolution order:

Try document.querySelector(containerSelector). If found, use textOffset and selectedText.length to locate the range within that container's text nodes and wrap it.
If the selector fails, search the full document for contextBefore + selectedText + contextAfter; if found, wrap the middle portion.
If both fail, log a console.warn with the annotation ID and skip silently (do not error).

Highlight rendering. Wrap the matched range in <span class="marginote-highlight" data-annotation-id="…"> with background-color set to the annotation's project color at 35% opacity. Use the marginote- class prefix and scope all styles in content.css.
Delete via context menu. Register a context menu item "Delete Marginote annotation" via chrome.contextMenus.create in the background service worker, shown only when right-clicking on a marginote-highlight element (use documentUrlPatterns and pass the annotation ID via a message from the content script to the background when the menu fires). On click, delete the annotation and unwrap the highlight span in-place. Do not override the browser's native right-click menu.
Styling. The tooltip is a small rounded rectangle with subtle shadow, ~32px tall, inside Shadow DOM so host styles can't affect it. The highlight span uses only background-color and border-radius: 2px to minimize layout disruption.
Verify on the three test site types and log results in TESTING.md.

3. Building on the existing Marginote extension, build out the full dashboard page.
Layout. Sidebar (left, ~240px) listing all projects with their color dots. Main area shows the selected project's annotations, organized by subgroup with an "Ungrouped" section at the top. The sidebar also has a "Search" item at the top and an "All annotations (timeline)" item (timeline view is added in Prompt 6 — leave a stub for now).
Annotation cards. Each card shows: the highlighted text (with a 3-line clamp and "Show more" toggle), the optional note, the page title as a clickable link (target="_blank"), the subgroup name as a small pill, and the relative date ("3 days ago") with the absolute date on hover.
Search. A search bar at the top of the main area. Scope: the currently selected project by default, with a checkbox "Search all projects." Matching is case-insensitive substring across selectedText, note, and pageTitle. Results show project + subgroup indicators.
Drag and drop. Native HTML5 drag and drop. Users can drag annotations:

Between subgroups within the same project — always allowed.
Between projects — allowed, but dropping onto a project in the sidebar shows a brief confirmation inline ("Move to <project>?" with Confirm / Cancel) because it's a bigger change.
When moved across projects, the annotation's subgroupId resets to null.

Markdown export. Per-project "Export to Markdown" button generates a single .md file for that project via a Blob download. Structure:
# <Project name>

_Exported <date>_

## <Subgroup name>

- > <highlighted text>
  <note if any>
  Source: [<page title>](<page url>) — <date saved>

## Ungrouped

- > …
Also add an "Export all (JSON)" button in the sidebar footer that dumps the entire storage state (projects, subgroups, annotations, schemaVersion) as a downloadable JSON file, and an "Import from JSON" button that reads a file, validates schemaVersion, runs migrate() if needed, and merges into the current storage (skipping duplicates by id, prompting on conflict).
Styling. Clean, Notion/Obsidian-inspired. Sidebar with subtle borders and hover states. Respect prefers-color-scheme for light/dark.
Verify drag-and-drop between subgroups and across projects, markdown export, and JSON round-trip (export → clear storage → import → everything restored).
